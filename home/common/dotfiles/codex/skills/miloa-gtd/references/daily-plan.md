# miloa タスクによる日次計画

最初に `gtd.md` を読む。日次計画では、GTD における `Next` の意味を保たなければならない。すべての `Next` タスクを今日のリストに平坦化してはならない。

## 確認する入力

1. 今日の日付。
2. **Google Calendar の今日の予定** - `list_events` または `search_entries` で取得し、**各イベントのタイトル・開始時刻・終了時刻・場所/種別の手がかり**を確認する。タイトルが取得できていない場合は、予定分析を進める前に取り直す。
3. 今日の `DailyNote` エントリ。存在する場合。
4. 未完了の `Task` エントリ。
5. `DueDate` が近いタスク。
6. `ScheduledDate` が今日以前のタスク。
7. `Task::Bucket = Next` のタスクと、その `ActivatedDate`。
8. `Task::Bucket = Await` のタスクのうち、`WaitingSince` が古いもの、または `ScheduledDate` が今日以前のもの。
9. Task に `EstimatedMinutes` がある場合はその値。
10. 関連するアクティブな `Project` エントリと、必要に応じてその `# Plan` セクション。
11. 今日の `Event` エントリ。予定の負荷が利用可能時間に影響する場合。

追加で、日次計画の前に以下を確認する：

1. **今日は大学登校日か**。平日はまず登校日である可能性を考え、授業間の休み時間を作業時間として安易に数えない。
2. **外出予定に移動時間が含まれているか**。Google Calendar に外出予定があるのに移動時間が明記されていなければ、移動時間を必ずユーザーに確認する。
3. **ご飯・洗濯・お風呂の時間が必要か**。カレンダーや既存計画から読めない場合は、必要の有無と概算時間をユーザーに確認する。
4. **希望する起床時刻・就寝時刻**。大学登校日でなければ必ず把握する。大学登校日で、起床後すぐ登校する前提が明確なら起床時刻の確認は省略可。
5. **戦略Area候補があるか**。`Area::StrategicBlock = true` の Area（戦略Area）に紐づく `Next` から、その日の枠を置けるかを確認する。日次計画では、締切タスクの詰め込みより前に戦略Area枠を検討する。

上記の確認が欠けている場合は、DailyNote を確定する前にユーザーへ短く質問して条件を埋める。

`EstimatedMinutes` がある場合も、それを絶対値として扱わない。日次計画では、見込み時間そのものよりも、今日の可処分時間に対して重すぎないか、短い枠に無理やり載せていないかを見る。

## Pre-flight Check (Stale Task Radar)

日次計画を開始する前に、**必ず以下の検索を実行し、該当タスクが存在するか確認する**。これらのタスクが存在する場合、**新規タスク選定や DailyNote 確定を行う前に処遇を決定する**。

### 検索クエリ（毎日必須）

1. **古い Next**: `Task::Bucket = Next` かつ `Status != Done` かつ `ActivatedDate` が3日以上前
2. **古い Await**: `Task::Bucket = Await` かつ `Status != Done` かつ `WaitingSince` が3日以上前
3. **期限超過**: `DueDate < today` かつ `Status != Done`
4. **予定超過**: `ScheduledDate <= today` かつ `Status != Done`

### 処遇決定フロー

該当タスクが1件でも存在する場合、以下の順で対応する：

1. **期限超過（DueDate < today）**
   - 最優先で扱う。即座に今日の Deep Work または Shallow Work 枠に割り当てる。
   - どうしても今日実行不可能な場合のみ、**理由を DailyNote に記録した上で** `DueDate` を再設定する。

2. **予定超過（ScheduledDate <= today）**
   - 今日のスケジュールに含めるか、**意図的に延期するか**を決定する。
   - 延期する場合は、**新しい `ScheduledDate` と延期理由を DailyNote に記録する**。
   - 無視して選定しないことは許容しない。

3. **古い Next（ActivatedDate >= 3日前）**
   - 3-6日: レビュー対象として扱い、今日実行するか、`ScheduledDate` を設定するか、分割・再設計するかを決定。
   - 7-13日: 実行、分解、`ScheduledDate` 設定、`Someday` 化、`Await` 化のいずれかを必ず行う。
   - 14日以上: 原則としてそのまま `Next` に残さない。`Someday` 化、別タスクへの分解、または削除を検討。

4. **古い Await（WaitingSince >= 3日前）**
   - 3-6日: 状況確認候補。ユーザーに「まだ待っているか」「催促が必要か」を確認。
   - 7-13日: 催促・再連絡・別経路確認を検討。`ScheduledDate` に催促日を設定するか、別ルートの `Next` を作成。
   - 14日以上: 待ち続けるだけにせず、Project 方針の見直し、代替アクションの検討、または `Someday` 化。

### Pre-flight 完了基準

以下がすべて満たされるまで、新規タスク選定に進まない：
- [ ] 期限超過タスクが0件、または全てに今日の処遇が決定済み
- [ ] 予定超過タスクが0件、または全てに今日の処遇（実行 or 延期理由記録）が決定済み
- [ ] 古い `Next` / `Await` が0件、または全てに具体的なアクション（実行/分割/日付変更/Bucket変更）が決定済み

**重要**: このチェックは「一度だけのレビュー」ではなく、**毎日の日次計画の最初のステップ**である。昨日チェックしたから今日スキップしてはならない。

## 戦略Area枠

日次計画では、通常タスクの選定前に、`Area::StrategicBlock = true` の Area（戦略Area）に紐づく `Project` のための時間枠を先に検討する。

- 対象は `StrategicBlock = true` の Area に紐づく `Project` のみとする。
- 候補は、原則として「今すぐ前提なしで進められる `Next`」から選ぶ。
- 適切な `Next` がない場合は、無理に枠を埋めず、Project の粒度見直しや `Next` 整備を優先候補として扱う。
- 戦略Area枠は、原則として **30〜90分** のまとまった時間で扱う。短い隙間に無理やりねじ込まない。
- その日に実質稼働可能時間の余裕があり、他の高優先タスクを圧迫しないなら、**90分を超える長めの枠**を置いてよい。特に設計・実装・調査など、切れ目なく進めた方が価値が高い作業では保守的に短く刻みすぎない。
- **ユーザーが拒否しない限り、毎日確保する**。各戦略Areaから最低1件ずつ選び、締切タスクを機械的に先詰めする前に先に予約する。
- 現実的でない場合は、「今日は省略する」ではなく「○曜日と△曜日に集中して確保する」など、**特定の日に徹底する案をユーザーに提案する**。
- 枠を置かなかった場合は、締切超過リスク、実働時間不足、前提未解決などの理由を短く明示する。
- 大学登校日では、戦略Area枠を授業間の細切れ時間に置かず、帰宅後や長めの連続ブロックを優先する。

## 稼働可能時間の計算とタイムスケジュール作成

Google Calendar の予定から実際の稼働可能時間を割り出し、**タスクを具体的な時間枠に配置する**：

### ステップ1: 予定の取得と時間マップ作成

1. **予定の取得**: Google Calendar API または MCP ツールで今日の予定を取得し、各イベントについてタイトルを必ず保持する
2. **タイムラインの作成**:
   - 大学登校日でなければ、ユーザーの希望する起床時刻・就寝時刻から計画対象時間帯を設定する
   - 大学登校日なら、登校・授業・帰宅を中心に時間帯を構成し、授業間の短い休み時間はまず移動/準備バッファとして置く
   - 固定予定を配置（会議、打ち合わせ、移動など）
   - ご飯・洗濯・お風呂について、必要と確認できたものを先にブロックする
   - 昼休憩を確保（最低1時間、できれば固定時間帯）
   - **空き時間ブロックを特定**

外出イベントで移動時間の明示がない場合は、そこで止まってユーザーに確認する。勝手に短く見積もって詰め込まない。

### ステップ2: 空き時間ブロックの評価

各空き時間ブロックを評価：

| 時間帯 | 長さ | 適した作業 | 理由 |
|--------|------|-----------|------|
| 9:00-10:00 | 60分 | Shallow Work | 短いブロック |
| 11:00-12:00 | 60分 | Deep Work | 午後の予定前、集中できる |
| 16:00-18:00 | 120分 | Deep Work | 連続ブロック、最長 |

評価基準を追加する：

- **大学登校日の授業間ギャップ**: たとえ 15〜30 分空いていても、原則は移動・準備・小休憩。タスク枠としては扱わない。
- **大学登校日の長めの空きコマ**: 60 分以上あっても、移動先・着席可能性・食事必要性が不明なら即座に Deep Work 枠にしない。キャンパス内で安定して着手できる短い Shallow Work か、休憩/食事バッファとして保守的に扱う。
- **予定直後/直前のギャップ**: 前後に移動や準備が必要なら、その分を差し引いてから可処分時間を出す。
- **場所依存**: PC 作業や資料読みに必要な環境がない可能性があるギャップは、実働時間として過大評価しない。
- **認知負荷**: 細切れの 10〜20 分枠は、タスク消化より回復や準備に向くことが多い。無理に埋めない。

### ステップ3: タスクの時間割り当て

**割り当ての原則**:

0. **戦略Area枠を先に置く**:
   - `StrategicBlock = true` の Area から、その日進める候補を各戦略Area最低1件ずつ先に選ぶ
   - まずは30〜90分を目安にまとまった枠を先に予約する
   - 余裕がある日なら、必要に応じて90分超の連続枠に拡張してよい
   - 締切超過リスクが極端に高い日を除き、残り時間に押し込めない

1. **Deep Work は大きなブロックに配置**:
   - 90-120分必要なら、連続した120分ブロックに配置
   - 午前中（脳のエネルギーが高い時間帯）を優先
   - 予定の直前には配置しない（準備・移動時間を確保）

2. **Shallow Work は予定間の隙間に配置**:
   - 30-60分のブロックに30分以下のタスクを詰め込む
   - 予定の直前・直後に配置可能
   - 同種の Shallow Work をバッチ処理

3. **優先順位で配置**:
   - Eisenhower 1（重要かつ緊急）→ 最も確実な時間帯
   - Deep Work（重要）→ 次に良い時間帯
   - Shallow Work（緊急）→ 残りの隙間

4. **EstimatedMinutes は弱い制約として使う**:
   - `15` や `30` は、短い Shallow Work や短時間の具体行動に向く目安として扱う
   - `60` 以上は、まとまった枠が必要かを確認するシグナルとして扱う
   - `240` は、そのまま4時間確保するより、今日着手する部分を切り出せないか先に検討する
   - 見込み時間が長くても、今日は一部だけ進める計画でよい

### ステップ4: 実質稼働時間の算出

```
標準稼働：9時間（9:00-18:00）
固定予定：
  - 10:00-11:00 打ち合わせ（1時間）
  - 14:00-15:30 会議（1.5時間）
  - 15:30-16:00 移動（0.5時間）
昼休憩：12:00-13:00（1時間）
バッファ：30分

実質稼働可能時間 = 9 - 3 - 1 - 0.5 = 4.5時間
```

ここでいう固定予定には、会議だけでなく、確認済みの移動時間、必要な食事、洗濯、お風呂も含める。

### ステップ5: タイムスケジュールの検証

割り当てたタスクの合計時間が実質稼働可能時間を超えていないか確認：
- Deep Work 合計 + Shallow Work 合計 ≤ 実質稼働可能時間
- 各時間帯にオーバーブッキングしていないか確認
- 予定間に最低15分のバッファがあるか確認

## 選定とスケジューリングの原則

- **実質稼働可能時間を上限**としてタスクを選定する。
- **選定したタスクには必ず時間枠を割り当てる** - 「いつやるか」が不明確なタスクは選定しない。
- **戦略Area枠を、通常タスク選定より先に検討する**。
- `EstimatedMinutes` は、詰め込み防止や枠の大きさ判断には使ってよいが、厳密な消化義務として扱わない。
- すべての `Next` タスクを機械的に含めない。
- `ActivatedDate` が古い `Next` は、そのまま今日の実行候補に入れるのではなく、粒度・前提・コミットの破綻を疑って扱う。
- 厳格な締切、期限超過または今日にスケジュールされたタスク、Project の進行を解除する真の Next アクションを優先する。
- 重い作業を詰め込みすぎない。
- 見込み時間が長い Task は、「今日はどこまで進めるか」を明示して部分着手に分けてよい。
- 直列型 Project では、前提タスクが完了する前に下流の作業を含めない。
- 外部待ちや条件が不明確なものは、実行作業ではなく「確認／待機」として扱う。特に古い `Await` は、催促・再確認・別ルート確認などの短いフォロー行動が必要かを判断する。
- ユーザーの予定が不明な場合でも、利用可能時間を過大評価しない。
- 平日の大学登校日は、授業間の隙間を「使えたらラッキー」ではなく「基本は使えない」に寄せて見積もる。
- 移動・生活時間が不明な外出日には、確認なしで最適化しない。

### Promise Lock（日付の遵守ルール）

`ScheduledDate` と `DueDate` は**「優先度マーカー」ではなく「約束」**として扱う。

**Promise First ルール**:
1. `ScheduledDate <= today` または `DueDate <= today` の未完了タスクがある場合、それらを**今日のスケジュールに含めるか、意図的に延期するか**を最初に決定する。
2. これらのタスクを「都合で後回し」にして無視することは許容しない。

**Escalation ルール**:
- `DueDate < today`（超過）のタスクがあれば、**最優先**で扱う。理由がなければ即座に今日の Deep Work 枠に割り当てる。
- `ScheduledDate < today`（超過）のタスクがあれば、実行または即座の再設定を行う。

**Audit Trail（監査証跡）**:
- `ScheduledDate` や `DueDate` を変更した場合は、**変更理由を DailyNote の「日次振り返り」セクションに必ず記録する**。
- 延期理由の例: 「大学授業延長のため」「前提タスクの未完了のため」「体調不良のため」
- 翌日以降も「昨日なぜ延期したか」を参照し、繰り返し延期のパターンを検出する。

**繰り返し延期の防止**:
- 同じタスクが2回以上延期された場合は、Task の粒度、前提条件、または `EstimatedMinutes` の見直しを優先する。
- 単なる「後回し」ではなく、**実行不可能な理由を特定して解消する**。

## 古い Next / Await の扱い

- `ActivatedDate` から 3 日以上経った `Next` は、そのまま今日の候補に入れる前にレビュー対象として扱う。
- `ActivatedDate` から 7 日以上経った `Next` は、実行するか、分解するか、`ScheduledDate` を置くか、`Someday` / `Await` に戻すかを先に判断する。
- `ActivatedDate` から 14 日以上経った `Next` は、原則そのまま温存しない。
- `WaitingSince` から 3 日以上経った `Await` は、状況確認候補として扱う。
- `WaitingSince` から 7 日以上経った `Await` は、催促・再連絡・別経路確認を検討する。
- `WaitingSince` から 14 日以上経った `Await` は、待ち続けるだけにせず、Project 方針や代替アクションを見直す。
- レビューで見送るだけなら、`ActivatedDate` / `WaitingSince` は更新しない。

## Deep Work / Shallow Work

Cal Newport の区別に従ってタスクを分類する。

### Deep Work

読書、設計、執筆、スライド作成、レビュー反映、難しい判断、長時間の集中が必要な作業に使用する。

- 1日あたり Deep Work は原則1〜2件までにする（実質稼働可能時間が4時間未満の場合は1件のみ）。
- 戦略Area枠を置く場合は、まずその枠を Deep Work 候補として評価する。
- 90〜120分のブロックを優先する。
- **Google Calendar の空き時間ブロックを確認**し、Deep Work に最適な時間帯（通常は午前または予定の少ない連続ブロック）を選定する。
- 大学登校日のキャンパス滞在中は、場所と連続性が明確でない限り Deep Work を置かない。
- **必ず具体的な時刻を割り当てる**（例：「9:00-11:00 設計ドキュメント作成」）
- 複数の Deep Work を細かい時間枠に分断しない。
- 重要な Deep Work がある日に、浅い達成感のための Shallow Work で予定を水増ししない。

### Shallow Work

日程調整、メッセージ、確認、ファイル整理、短い提出、軽いメモ、単純な反復作業に使用する。

- 5〜30分の枠にまとめて処理する。
- **予定間の隙間に配置**（例：「10:00-10:15 メールチェック」「15:30-16:00 書類整理」）ただし大学登校日の授業間休みは原則除く。
- Deep Work の前に Shallow Work がだらだら広がらないようにする。
- ブロッカーを解除する Shallow Work は優先する。

## Eisenhower トリアージ

| 領域 | 意味 | 扱い |
|---|---|---|
| 1 | 重要かつ緊急 | 今日の最有力候補 |
| 2 | 重要だが緊急ではない | Deep Work 候補。ただし数は絞る |
| 3 | 緊急だが重要度は低い | Shallow Work として短時間で処理 |
| 4 | 重要でも緊急でもない | 今日はやらない |

## DailyNote 要件

簡潔なチャット要約を返し、同じ計画を今日の `DailyNote` に書き込む。今日の DailyNote が存在する場合は、適切なセクションに追記または更新する。存在しない場合は、`DailyNote::Date` を今日に設定して作成する。

推奨 DailyNote 形式：

```md
## 今日の稼働時間
- 実質稼働可能時間: X時間
- Google Calendar 予定:
  - 10:00-11:00 打ち合わせ
  - 14:00-15:30 会議
- 戦略Area枠:
  - 9:30-10:30 戦略Areaの要件整理

## タイムスケジュール

| 時間 | 予定 / タスク | 種別 | 備考 |
|------|--------------|------|------|
| 9:00-9:30 | メール・Slack確認 | Shallow | 朝の情報収集 |
| 9:30-10:30 | 戦略Areaの要件整理 | Deep | 戦略Area枠 |
| 10:30-11:00 | 打ち合わせ準備 | Shallow | 会議前の調整 |
| 11:00-12:00 | 打ち合わせ | 固定 | Google Calendar |
| 12:00-13:00 | 昼休憩 | - | - |
| 13:00-14:00 | コードレビュー | Deep | 午後の最初の集中 |
| 14:00-15:30 | 定例会議 | 固定 | Google Calendar |
| 15:30-16:00 | 承認依頼の確認 | Shallow | 会議後の小タスク |
| 16:00-17:30 | リファクタリング | Deep | 最後の集中ブロック |
| 17:30-18:00 | 明日の準備・振り返り | Shallow | 日次クロージング |

## 今日の方針
- 最優先は戦略Area枠として置いた「戦略Areaの要件整理」。
- 会議は2つあるため、Deep Work は戦略枠1件と通常集中作業1件に絞る。
```

## 作業終了時刻の確認

日次計画の前に、その日の**作業終了時刻**をユーザーに確認する。これは睡眠時間と翌日の状態を左右する最重要パラメータである。

### 原則的な目安

- **月〜木**: 27:00〜28:00（深夜3〜4時）までの作業を許容する。ただし翌日の予定と体調を確認。
- **金〜日**: 23:00頃までに終えられることが理想。週末の回復を優先。
- **深夜遅くまで外出がある日**: 帰宅後の作業時間は確保できない前提とし、帰宅後は就寝準備のみにする。

### 確認の仕方

1. 「今日は何時まで作業する予定ですか？」と直接尋ねる。
2. 上記の目安と乖離が大きい場合は、翌日への影響を簡潔に提醒する。
3. 作業終了時刻を確認した上で、**その時間までの可処分時間を計算し直す**。

**重要**: 作業終了時刻は「できれば」ではなく、その日の**計算の前提**である。終了時刻が不明確なままタスク選定を進めない。

最終基準：今日のスケジュールは、現実的に実行可能な粒度で、**いつ何をやるかが明確**であるべき。
