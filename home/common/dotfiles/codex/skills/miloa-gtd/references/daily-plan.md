# miloa タスクによる日次計画

最初に `gtd.md` を読む。日次計画では、GTD における `Next` の意味を保たなければならない。すべての `Next` タスクを今日のリストに平坦化してはならない。

## 確認する入力

1. 今日の日付。
2. **Google Calendar の今日の予定** - `list_events` または `search_entries` で取得し、時間ブロックを確認する。
3. 今日の `DailyNote` エントリ。存在する場合。
4. 未完了の `Task` エントリ。
5. `DueDate` が近いタスク。
6. `ScheduledDate` が今日以前のタスク。
7. `Task::Bucket = Next` のタスク。
8. 関連するアクティブな `Project` エントリと、必要に応じてその `# Plan` セクション。
9. 今日の `Event` エントリ。予定の負荷が利用可能時間に影響する場合。

## 稼働可能時間の計算とタイムスケジュール作成

Google Calendar の予定から実際の稼働可能時間を割り出し、**タスクを具体的な時間枠に配置する**：

### ステップ1: 予定の取得と時間マップ作成

1. **予定の取得**: Google Calendar API または MCP ツールで今日の予定を取得
2. **タイムラインの作成**:
   - 標準稼働時間帯を設定（例：9:00-18:00）
   - 固定予定を配置（会議、打ち合わせ、移動など）
   - 昼休憩を確保（最低1時間、できれば固定時間帯）
   - **空き時間ブロックを特定**

### ステップ2: 空き時間ブロックの評価

各空き時間ブロックを評価：

| 時間帯 | 長さ | 適した作業 | 理由 |
|--------|------|-----------|------|
| 9:00-10:00 | 60分 | Shallow Work | 短いブロック |
| 11:00-12:00 | 60分 | Deep Work | 午後の予定前、集中できる |
| 16:00-18:00 | 120分 | Deep Work | 連続ブロック、最長 |

### ステップ3: タスクの時間割り当て

**割り当ての原則**:

1. **Deep Work は大きなブロックに配置**:
   - 90-120分必要なら、連続した120分ブロックに配置
   - 午前中（脳のエネルギーが高い時間帯）を優先
   - 予定の直前には配置しない（準備・移動時間を確保）

2. **Shallow Work は予定間の隙間に配置**:
   - 30-60分のブロックに30分以下のタスクを詰め込む
   - 予定の直前・直後に配置可能
   - 同種の Shallow Work をバッチ処理

3. **優先順位で配置**:
   - Eisenhower 1（重要かつ緊急）→ 最も確実な時間帯
   - Deep Work（重要）→ 次に良い時間帯
   - Shallow Work（緊急）→ 残りの隙間

### ステップ4: 実質稼働時間の算出

```
標準稼働：9時間（9:00-18:00）
固定予定：
  - 10:00-11:00 打ち合わせ（1時間）
  - 14:00-15:30 会議（1.5時間）
  - 15:30-16:00 移動（0.5時間）
昼休憩：12:00-13:00（1時間）
バッファ：30分

実質稼働可能時間 = 9 - 3 - 1 - 0.5 = 4.5時間
```

### ステップ5: タイムスケジュールの検証

割り当てたタスクの合計時間が実質稼働可能時間を超えていないか確認：
- Deep Work 合計 + Shallow Work 合計 ≤ 実質稼働可能時間
- 各時間帯にオーバーブッキングしていないか確認
- 予定間に最低15分のバッファがあるか確認

## 選定とスケジューリングの原則

- **実質稼働可能時間を上限**としてタスクを選定する。
- **選定したタスクには必ず時間枠を割り当てる** - 「いつやるか」が不明確なタスクは選定しない。
- すべての `Next` タスクを機械的に含めない。
- 厳格な締切、期限超過または今日にスケジュールされたタスク、Project の進行を解除する真の Next アクションを優先する。
- 重い作業を詰め込みすぎない。
- 直列型 Project では、前提タスクが未完了の間は下流の作業を含めない。
- 外部待ちや条件が不明確なものは、実行作業ではなく「確認／待機」として扱う。
- ユーザーの予定が不明な場合でも、利用可能時間を過大評価しない。

## Deep Work / Shallow Work

Cal Newport の区別に従ってタスクを分類する。

### Deep Work

読書、設計、執筆、スライド作成、レビュー反映、難しい判断、長時間の集中が必要な作業に使用する。

- 1日あたり Deep Work は原則1〜2件までにする（実質稼働可能時間が4時間未満の場合は1件のみ）。
- 90〜120分のブロックを優先する。
- **Google Calendar の空き時間ブロックを確認**し、Deep Work に最適な時間帯（通常は午前または予定の少ない連続ブロック）を選定する。
- **必ず具体的な時刻を割り当てる**（例：「9:00-11:00 設計ドキュメント作成」）
- 複数の Deep Work を細かい時間枠に分断しない。
- 重要な Deep Work がある日に、浅い達成感のための Shallow Work で予定を水増ししない。

### Shallow Work

日程調整、メッセージ、確認、ファイル整理、短い提出、軽いメモ、単純な反復作業に使用する。

- 5〜30分の枠にまとめて処理する。
- **予定間の隙間に配置**（例：「10:00-10:15 メールチェック」「15:30-16:00 書類整理」）
- Deep Work の前に Shallow Work がだらだら広がらないようにする。
- ブロッカーを解除する Shallow Work は優先する。

## Eisenhower トリアージ

| 領域 | 意味 | 扱い |
|---|---|---|
| 1 | 重要かつ緊急 | 今日の最有力候補 |
| 2 | 重要だが緊急ではない | Deep Work 候補。ただし数は絞る |
| 3 | 緊急だが重要度は低い | Shallow Work として短時間で処理 |
| 4 | 重要でも緊急でもない | 今日はやらない |

## DailyNote 要件

簡潔なチャット要約を返し、同じ計画を今日の `DailyNote` に書き込む。今日の DailyNote が存在する場合は、適切なセクションに追記または更新する。存在しない場合は、`DailyNote::Date` を今日に設定して作成する。

推奨 DailyNote 形式：

```md
## 今日の稼働時間
- 実質稼働可能時間: X時間
- Google Calendar 予定:
  - 10:00-11:00 打ち合わせ
  - 14:00-15:30 会議

## タイムスケジュール

| 時間 | 予定 / タスク | 種別 | 備考 |
|------|--------------|------|------|
| 9:00-9:30 | メール・Slack確認 | Shallow | 朝の情報収集 |
| 9:30-11:00 | :entry{#<task_id>} 設計ドキュメント作成 | Deep | 会議前に集中作業 |
| 11:00-12:00 | 打ち合わせ | 固定 | Google Calendar |
| 12:00-13:00 | 昼休憩 | - | - |
| 13:00-14:00 | :entry{#<task_id>} コードレビュー | Deep | 午後の最初の集中 |
| 14:00-15:30 | 定例会議 | 固定 | Google Calendar |
| 15:30-16:00 | :entry{#<task_id>} 承認依頼の確認 | Shallow | 会議後の小タスク |
| 16:00-17:30 | :entry{#<task_id>} リファクタリング | Deep | 最後の集中ブロック |
| 17:30-18:00 | 明日の準備・振り返り | Shallow | 日次クロージング |

## 今日の方針
- 最優先は :entry{#<task_id>}。
- 会議は2つあるため、Deep Workは3ブロックに分けて確保。

## タスク詳細

### 9:30-11:00 :entry{#<entry_id>}
- 種別: Deep Work
- Eisenhower: 重要 / 緊急ではない
- 理由: Project Xの設計フェーズ。締切は今週末。
- 目安: 90分
- 完了条件: 設計図の初稿が完成し、レビュー依頼可能な状態

### 13:00-14:00 :entry{#<entry_id>}
- 種別: Deep Work
- Eisenhower: 重要かつ緊急
- 理由: 明日のリリースに必要
- 目安: 60分
- 完了条件: PR承認が得られる

### 15:30-16:00 :entry{#<entry_id>}
- 種別: Shallow Work
- Eisenhower: 緊急 / 重要度低
- 理由: 承認待ちタスクのステータス確認
- 目安: 30分
- 完了条件: 承認状況の把握、必要ならフォロー

## 今日やらない
- :entry{#<entry_id>}: 理由（時間がない、前提タスク未完了など）
```

**スケジュール作成の検証リスト**:
- [ ] すべてのタスクに具体的な時間枠が割り当てられているか
- [ ] Deep Work は十分な長さの連続ブロックに配置されているか（90分推奨）
- [ ] 予定間に最低15分のバッファがあるか
- [ ] タスク合計時間 ≤ 実質稼働可能時間
- [ ] 急な対応用に30分程度の余裕があるか
- [ ] 最後のブロックは18:00前に終わるか（オーバーラン防止）

## 禁止事項

* すべての `Next` タスクを今日のリストとして使用する。
* **時間枠の割り当てなしにタスクを選定する**（「いつやるか」が不明確なタスクは選定しない）。
* Deep Work を軽率に3件以上追加する。
* 好みだけを理由に、締切が近い作業を先延ばしにする。
* 前提タスクが完了する前に、Project の将来ステップを追加する。
* DailyNote への書き込みを忘れる。
* 新しいタグやフィールドを気軽に追加する。
* DailyNote を長い理論で膨らませる。

最終基準：今日のスケジュールは、現実的に実行可能な粒度で、**いつ何をやるかが明確**であるべき。
