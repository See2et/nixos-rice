# miloa タスクによる日次計画

最初に `gtd.md` を読む。日次計画では、GTD における `Next` の意味を保たなければならない。すべての `Next` タスクを今日のリストに平坦化してはならない。

## 確認する入力

1. 今日の日付。
2. 今日の `DailyNote` エントリ。存在する場合。
3. 未完了の `Task` エントリ。
4. `DueDate` が近いタスク。
5. `ScheduledDate` が今日以前のタスク。
6. `Task::Bucket = Next` のタスク。
7. 関連するアクティブな `Project` エントリと、必要に応じてその `# Plan` セクション。
8. 今日の `Event` エントリ。予定の負荷が利用可能時間に影響する場合。

## 選定原則

- すべての `Next` タスクを機械的に含めない。
- 厳格な締切、期限超過または今日にスケジュールされたタスク、Project の進行を解除する真の Next アクションを優先する。
- 重い作業を詰め込みすぎない。
- 直列型 Project では、前提タスクが未完了の間は下流の作業を含めない。
- 外部待ちや条件が不明確なものは、実行作業ではなく「確認／待機」として扱う。
- ユーザーの予定が不明な場合でも、利用可能時間を過大評価しない。

## Deep Work / Shallow Work

Cal Newport の区別に従ってタスクを分類する。

### Deep Work

読書、設計、執筆、スライド作成、レビュー反映、難しい判断、長時間の集中が必要な作業に使用する。

- 1日あたり Deep Work は原則1〜2件までにする。
- 90〜120分のブロックを優先する。
- 複数の Deep Work を細かい時間枠に分断しない。
- 重要な Deep Work がある日に、浅い達成感のための Shallow Work で予定を水増ししない。

### Shallow Work

日程調整、メッセージ、確認、ファイル整理、短い提出、軽いメモ、単純な反復作業に使用する。

- 5〜30分の枠にまとめて処理する。
- Deep Work の前に Shallow Work がだらだら広がらないようにする。
- ブロッカーを解除する Shallow Work は優先する。

## Eisenhower トリアージ

| 領域 | 意味 | 扱い |
|---|---|---|
| 1 | 重要かつ緊急 | 今日の最有力候補 |
| 2 | 重要だが緊急ではない | Deep Work 候補。ただし数は絞る |
| 3 | 緊急だが重要度は低い | Shallow Work として短時間で処理 |
| 4 | 重要でも緊急でもない | 今日はやらない |

## DailyNote 要件

簡潔なチャット要約を返し、同じ計画を今日の `DailyNote` に書き込む。今日の DailyNote が存在する場合は、適切なセクションに追記または更新する。存在しない場合は、`DailyNote::Date` を今日に設定して作成する。

推奨 DailyNote 形式：

```md
## 今日の方針
- 最優先はXXX。

## 今日のタスク

### 1. :entry{#<entry_id>}
- 種別: Deep Work / Shallow Work
- Eisenhower: 重要 / 緊急
- 理由: 締切、ScheduledDate、Project上の位置など
- 目安: XX分
- 完了条件: ...

### 2. :entry{#<entry_id>}
- 種別:
- Eisenhower:
- 理由:
- 目安:
- 完了条件:

## 今日やらない
- :entry{#<entry_id>}: 理由
````

## 禁止事項

* すべての `Next` タスクを今日のリストとして使用する。
* Deep Work を軽率に3件以上追加する。
* 好みだけを理由に、締切が近い作業を先延ばしにする。
* 前提タスクが完了する前に、Project の将来ステップを追加する。
* DailyNote への書き込みを忘れる。
* 新しいタグやフィールドを気軽に追加する。
* DailyNote を長い理論で膨らませる。

最終基準：今日のリストは、現実的に完了可能であるべき。
