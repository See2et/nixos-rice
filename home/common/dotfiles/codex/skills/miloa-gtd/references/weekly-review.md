# miloa タスクによる週次レビュー

日次計画の上位互換。週の目標を立て、日々の計画が週全体の整合性を保つようにする。推奨実行タイミングは日曜または月曜。

## 書き込み前ゲート

以下が終わるまで `WeeklyNote` 作成や Task 更新をしない：

- `list_tags` を読んだ
- `references/query-pack.md` にある週次レビュー必須取得順を完了した
- Google Calendar の今週予定を取得した
- `Organization::WeeklyReview = true` の全 Organization を確認した

Google Calendar をまだ見ていない、または Organization レビューが終わっていない状態で週次計画本文を書き始めない。

## 週次レビューフロー

### ステップ1: 先週の振り返り

1. **戦略Organizationの進捗確認**
   - `Organization::StrategicBlock = true` の Organization の先週の目標と実績を比較
   - 先週の DailyNote を参照し、戦略Organization枠を何日置いたか、何件進めたかを確認
   - 進捗が計画通りでなかった場合、理由を特定（時間不足、前提未解決、見積もり過大など）

2. **締切タスクの実績確認**
   - 先週に `DueDate` が設定されていたタスクの完了率
   - 超過・延期が発生した場合、パターンを特定（予定過多、見積もり誤り、緊急対応など）

3. **古い Next / Await の一掃状況**
   - 先週の週次レビュー時に存在した古い `Next` / `Await` が解消されたか
   - 繰り返し滞留するタスクのパターンを検出

### ステップ2: 今週の全体把握

1. **今週の締切タスク洗い出し**
   - `DueDate` が今週内にある未完了タスクを全件リストアップ
   - 各タスクに必要な作業時間を概算し、週全体の負荷を把握

2. **今週の ScheduledDate 分布確認**
   - `ScheduledDate` が今週内にある未完了タスクをリストアップ
   - 曜日ごとの集中度を確認し、偏りがあれば調整

3. **固定予定の確認**
   - Google Calendar で今週の予定を確認
   - 大学登校日、会議、外出、締切などの固定要素をマッピング
   - Google Calendar を見ていない場合はここで停止する。miloa の `Event` や既存 `DailyNote` だけで代用しない

### ステップ3: 週間タイムライン作成

月〜日の各日に以下をマッピングする：

```md
## 今週のタイムライン

| 曜日 | 固定予定 | 締切タスク | ScheduledDate | 戦略枠候補 | 可処分時間 |
|------|---------|-----------|---------------|-----------|-----------|
| 月   | 授業A, B  | レポート締切 | 設計レビュー   | OrgA要件整理 | 2h        |
| 火   | 授業C     | -         | -             | OrgB実装     | 4h        |
| 水   | 打ち合わせ | -         | 承認依頼確認   | -           | 3h        |
| ...  | ...       | ...       | ...           | ...         | ...       |
```

**マッピングのルール**:
- 大学登校日は、授業間の隙間を「基本使えない」として扱い、帰宅後の時間を可処分時間とする
- 締切タスクは `DueDate` 当日ではなく、**完了に必要な作業日を逆算**して配置
- 戦略プロジェクト枠は、**可処分時間が2時間以上ある日**に優先して配置
- 可処分時間の計算は `daily-plan.md` の「稼働可能時間の計算」と同じ基準で行う

### ステップ4: 戦略Organizationの週次目標

1. **`Organization::StrategicBlock = true` の各 Organization について**:
   - 今週の目標成果を1文で定義（Organization の Description を参照し、重点目標と整合させる）
   - 目標達成に必要な最低限の作業日数と時間を算出
   - 今週のタイムラインにその日数・時間を配置できるか確認

2. **配置できない場合**:
   - 目標を縮小する（「要件整理の半分まで」）
   - または、他の予定の調整可能性をユーザーに確認
   - 無理に配置せず、**現実的な週次目標**を設定する

### ステップ5: WeeklyNote エントリの作成

週次レビューが完了したら、結果を miloa の `WeeklyNote` エントリとして記録する。

**WeeklyNote スキーマ**:
- `Tag`: `WeeklyNote`
- `StartDate`: その週の開始日
- `Title`: `YYYY-MM-DD 週次レビュー`（またはユーザーが指定する形式）

実際のフィールド名は `list_tags` を正とする。ここでの表記より実スキーマを優先する。

**推奨 WeeklyNote 本文**:
```md
## 先週の振り返り
- 戦略Organizationの進捗: ...
- 締切タスクの完了率: ...

## 今週のタイムライン
| 曜日 | 戦略OrgA | 戦略OrgB | 可処分時間 | 備考 |
|------|----------|----------|-----------|------|
| 月   | ○        | ○        | 4h        |      |
| 火   | ×        | ○        | 2h        | 予定多い |
| ...  | ...      | ...      | ...       | ...  |

## 今週の目標
- [Organization名]: 〇〇を完了させる
- [Organization名]: △△の設計を終わらせる

## 注意事項
- ...
```

**WeeklyNote の運用ルール**:
- 週次レビュー毎に**新規作成**する（上書きではなく、週の記録として残す）
- 日次計画時に、その週の WeeklyNote を参照し、週次目標とタイムラインに沿っているか確認する
- 週の途中で週次目標を変更した場合は、WeeklyNote に追記し変更理由を記録する
- 週次レビューのデフォルトは `WeeklyNote` 記録まで。Task の `ScheduledDate` / `DueDate` 更新は必要性が高いものだけ最小限に行う
- Task の日付を更新する場合は、WeeklyNote 本文に「なぜ今週その日に置いたか」を短く残す

### ステップ6: 日次計画への派生

週次レビューが完了したら、各日の日次計画は以下を満たすように作成する：

- その日の戦略プロジェクト枠が週次目標に向かっているか
- 締切タスクが週間タイムライン通りに進んでいるか
- `ScheduledDate` の過多や偏りがないか

## 週次レビューでの健全性チェック

週次レビュー時に以下を優先して検出する：

1. **締切の集中**: `DueDate` が同じ日または連続する日に複数集中していないか
2. **戦略Organizationの週次進捗ゼロ**: 今週のタイムラインに戦略Organization枠が1件もない場合、理由を明示する
3. **繰り返し延期パターン**: 先週から同じタスクが `ScheduledDate` を変更し続けていないか
4. **過積載週**: 可処分時間の合計に対して必要作業時間が明らかに超過していないか
5. **放置タスクの週次蓄積**: 週次レビュー時に7日以上古い `Next` / `Await` が複数残っていないか
6. **Google Calendar 未取得のまま計画していないか**

## 週次レビューの完了基準

- [ ] 先週の戦略Organization進捗を確認し、次週に反映すべき調整を特定した
- [ ] 今週の締切タスクを全件リストアップし、作業日を逆算して配置した
- [ ] 今週のタイムライン（月〜日）を作成し、各日の可処分時間を概算した
- [ ] `Organization::StrategicBlock = true` の各戦略Organizationの週次目標を設定した
- [ ] 週間タイムラインに戦略Organization枠を毎日配置し、現実的でない日には特定日集中案を記録した
- [ ] `Organization::WeeklyReview = true` の全Organizationをレビューした
- [ ] 締切集中、過積載、戦略枠ゼロなどの異常を検出し、調整方針を決定した
- [ ] Google Calendar を取得する前に週間タイムラインを書き始めていない
- [ ] **WeeklyNote エントリを作成した**
- [ ] 日次計画が週次タイムラインから派生する構造になっている

## 週次レビューと日次計画の関係

```
週次レビュー（日曜/月曜）
├── 週間タイムライン作成
├── 戦略プロジェクトの週次目標設定
└── 締切・ScheduledDate の週内配置
    ↓
日次計画（毎日）
├── Pre-flight Check（Stale Task Radar）
├── 週次タイムラインに基づくその日の戦略枠確認
└── タイムスケジュール作成
```

**原則**: 日次計画は週次計画の**派生**である。週次計画で決めた戦略枠や締切配置を、日次計画で無視して再構成しない。
